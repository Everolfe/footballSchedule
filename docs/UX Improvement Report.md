# UX Improvement Report

## Расписание футбольных матчей

---

# 1. Введение

В данном отчёте представлена оценка удобства использования и предложения по улучшению пользовательского опыта (UX) для системы `footballSchedule`. Так как система представляет собой набор микросервисов с REST API, под "пользователем" здесь понимается разработчик фронтенда или системный интегратор, который взаимодействует с API.

Анализ проведён на основе атрибутов качества группы "Usability" стандарта *ISO/IEC 25010:2011 (SQuaRE)*. Также учтены лучшие практики проектирования API, которые можно соотнести с рекомендациями:

* Microsoft Usability Guidelines
* Apple Human Interface Guidelines
* Google Material Design 
* WCAG 2.0 (Web Content Accessibility Guidelines)

Цель улучшений — сделать API более:

* Понятным и логичным.
* Лёгким для интеграции и отладки.
* Предсказуемым и безопасным в использовании.
* Информативным с точки зрения обратной связи.

---

# 2. Оценка по атрибутам юзабилити ISO/IEC 25010

## 2.1 Распознаваемость соответствия

**Определение:** Степень, с которой пользователь может понять, подходит ли ПО для его нужд.

### Текущее состояние (as is)

* Наличие микросервисов `Arena`, `Match`, `Player`, Team с соответствующими REST-эндпоинтами (например, `/api/players`) сразу даёт понять предметную область. Пользователь видит, что система управляет именно этими сущностями.
* Проблема: Отсутствие единой точки входа (API Gateway) и версионирования API. Разработчику нужно знать адреса всех 4 сервисов, чтобы начать работу.

### Предлагаемые улучшения:

* Внедрение API Gateway (как указано в плане рефакторинга). Единый хост и единые правила доступа.
* Добавление версионирования API (например, /api/v1/players). Это сразу сообщит пользователю о стабильности контракта.

---

## 2.2 Learnability

**Определение:** Степень, с которой пользователь может научиться эффективно использовать систему.

### Текущее состояние (as is):

* Использование стандартных CRUD-операций (GET, POST, PUT, DELETE) и понятных DTO (например, CreatePlayerDto) снижает порог входа.
* Наличие автоматической документации API (Swagger). Новому разработчику не придётся читать код контроллеров или внешнюю wiki, чтобы понять структуру запросов и ответов.

### Предлагаемые улучшения:

* Обогащение кода контроллеров комментариями для генерации более качественной документации (описание полей, коды ответов).

---

## 2.3 Используемость

**Определение:** Степень, с которой пользователь может управлять системой и контролировать её.

### Текущее состояние (as is):

* Чёткое разделение на микросервисы упрощает понимание границ ответственности.
* Проблема: Отсутствие единой точки входа (API Gateway) и единообразной обработки сквозных задач (CORS, логирование, безопасность) на всех сервисах. Разработчику нужно настраивать интеграцию с каждым сервисом по отдельности, что снижает операбельность.

### Предлагаемые улучшения:

* Внедрение API Gateway (Spring Cloud Gateway) для централизованной маршрутизации, применения CORS-политик, логирования и аутентификации. Это сделает управление трафиком прозрачным.
* Унификация формата ответов об ошибках (уже реализовано в GlobalExceptionHandler) — важный шаг, который нужно сохранить и расширить.

---

## 2.4 Защита от ошибок пользователя

**Определение:**  Степень, с которой система защищает пользователей от совершения ошибок.

### Текущее состояние (as is):

* Валидация на уровне DTO с помощью аннотаций (@NotNull, @Size) — это отличный механизм защиты от передачи некорректных данных.
* Централизованная обработка ошибок (GlobalExceptionHandler) возвращает понятные сообщения об ошибках валидации.
* Проблема: Отсутствие защитных механизмов на уровне инфраструктуры (таймауты, повторные попытки) для повышения отказоустойчивости.

### Предлагаемые улучшения:

* Внедрение `Resilience4j` (как в плане) для добавления таймаутов и повторных попыток (retry). Это защитит пользователя от временных сбоев и зависаний.
* Расширение валидации для более сложных бизнес-правил (например, дата матча не может быть в прошлом при создании).

---

## 2.5 Эстетика взаимодействия

**Определение:** Степень, с которой интерфейс (в данном случае API) выглядит приятно и организованно.

### Текущее состояние (as is):

* Использование DTO разделяет внутренние модели и внешние контракты, делая JSON-ответы чистыми и релевантными.
* Проблема: В ответах могут возвращаться поля, не нужные клиенту (например, технические id или временные метки в неотформатированном виде), что "засоряет" эстетику ответа.

### Предлагаемые улучшения:

* Тщательное проектирование DTO. Создавать отдельные DTO для запросов (*RequestDto) и ответов (*ResponseDto), включая только необходимые поля.
*Использование согласованных форматов (например, даты и время в ISO 8601: 2024-05-21T19:30:00Z). Это делает ответы предсказуемыми и "красивыми" для парсинга.

---

## 2.6 Доступность

Определение: Степень, с которой система может быть использована людьми с разными возможностями (для API — различными клиентами и в различных условиях).

### Текущее состояние (as is):

* Контейнеризация и Docker Compose обеспечивают лёгкий локальный запуск, что повышает доступность для разработки.
* Наличие мониторинга и метрик.
* Проблема: Отсутствие HTTPS делает трафик небезопасным.

### Предлагаемые улучшения:

* Настройка HTTPS на уровне API Gateway для защиты данных.

# 3. Уровни пользовательского опыта 

## 3.1 Уровень стратегии

**Цель системы:**

* Предоставить надёжный и удобный API для управления данными о футбольных соревнованиях (арены, матчи, игроки, команды).

**Ожидания пользователя (разработчика): **

* Простота интеграции, предсказуемость, хорошая документация и стабильность.

Предлагаемые улучшения полностью соответствуют этой стратегии: снижение когнитивной нагрузки через единый шлюз (Gateway) и автоматическую документирование (Swagger).

---

## 3.2 Уровень набора возможностей (Scope)

**Текущий набор:**  CRUD для 4 сущностей, поиск по ID и списков.

**Предлагается добавить:**

* Фильтрация и пагинация для списков (например, GET `/api/v1/players?teamId=1&page=0&size=10`).
* Поиск по определённым полям.
* Более сложные бизнес-запросы (например, расписание матчей команды на дату).
---

## 3.3 Уровень структуры

### Текущее состояние:

* Прямые вызовы к разным микросервисам (каждый на своём порту/адресе).

### Целевое состояние:

* Иерархическая структура через API Gateway.
* Чёткая структура URL: /api/v1/{resource}/{id}

---

## 3.4 Уровень компоновки (Skeleton)

### Текущее состояние

* Использование DTO, что является хорошей основой.

### Предлагаемые улучшения:

* Единый шаблон для ответов со списком: 
	* { "content": [...], "page": { "size": 10, "totalElements": 100 } }.
* Единый шаблон для ошибок:
	* { "timestamp": "...", "status": 400, "error": "Validation Failed", "details": ["field1: must not be null"] }

---

## 3.5 Уровень поверхности

### Текущее состояние

* JSON-ответы формируются автоматически.

### Целевое состояние:

* "Чистый" JSON без лишних полей.
* Красивая и интерактивная документация Swagger UI с примерами запросов и ответов.
* Использование стандартных HTTP-кодов (200, 201, 400, 404, 500) для понятности.

# 6.  Планируемые будущие улучшения UX

1. Унифицировать форматы DTO для всех ответов со списками.
2. Внедрить API Gateway (Spring Cloud Gateway).
3. Внедрить версионирование API.
4. Добавить аутентификацию (JWT).
5. Перейти к Database per Service, что улучшит отказоустойчивость и независимость сервисов, а значит, и стабильность API для пользователя.

# 7. Заключение

Проведённый анализ показал, что текущая архитектура API системы footballSchedule имеет хорошую основу (слоистость, DTO, валидация), но её пользовательский опыт для разработчика-интегратора может быть значительно улучшен.

Предложенные изменения, основанные на стандарте ISO 25010 и лучших практиках, направлены на то, чтобы сделать API более:

* Очевидным (единый Gateway, версионирование);
* Удобным (пагинация, фильтры);
* Предсказуемым (стандартные форматы ответов и ошибок);
* Надёжным (Resilience4j).
* Improved UI aesthetics
* Enhanced accessibility foundation

Реализация этих улучшений превратит набор микросервисов в стройную, профессиональную и "дружелюбную" к разработчику платформу.

# 9. Заключение по трассируемости

Представленная матрица трассировки демонстрирует, что все предложенные улучшения пользовательского опыта (UX) для API системы footballSchedule имеют обоснованную связь с:

* Конкретной проблемой в текущей реализации (as is), выявленной в ходе анализа архитектуры и кодовой базы.
* Чётким UX-решением, направленным на устранение этой проблемы.
* Конкретным атрибутом usability из стандарта ISO/IEC 25010:2011, что подтверждает системный подход к улучшениям, а не просто субъективные "косметические" изменения.

Такой подход гарантирует, что работа по улучшению UX является структурированной, измеримой и полностью интегрированной в общий процесс разработки и устранения технического долга. Каждое изменение может быть отслежено от первоначальной потребности пользователя (разработчика API) до его итоговой реализации в коде и инфраструктуре.
