## 1. Текущее состояние проекта

Проект представляет собой набор микросервисов (Arena, Match, Player, Team), реализующих REST API для работы с соответствующими сущностями. В проекте используется современный стек: Java 21, Spring Boot, PostgreSQL, Docker. Ниже приведён анализ текущего состояния по различным аспектам.

### 1.1 Архитектура
- **Слоистая структура:** Чёткое разделение на контроллеры, сервисы и репозитории внутри каждого микросервиса.
- **База данных:** Все сервисы используют единую PostgreSQL. Это упрощает поддержку транзакционной целостности и ссылочной целостности на ранних этапах, но в долгосрочной перспективе ограничивает независимость сервисов.
- **Кэширование:** Реализовано двухуровневое кэширование (Spring Cache + собственный кэш), что положительно сказывается на производительности.
- **Обработка ошибок:** Централизованная через `GlobalExceptionHandler`, возвращаются унифицированные ответы.

### 1.2 Код
- **Дублирование:** DTO, конфигурации CORS, валидация и маппинг повторяются в каждом микросервисе. Это увеличивает объём кода и стоимость внесения изменений.
- **Читаемость:** Базовая структура понятна, однако из-за дублирования общая картина может восприниматься сложнее.

### 1.3 Процессы разработки
- **Сборка и развёртывание:** Автоматизированы с помощью Docker и `docker-compose`. Локальный запуск всего стека выполняется одной командой.
- **Тестирование:** В проекте отсутствуют автоматические тесты (модульные, интеграционные). Это зона для развития, которая повысит уверенность при изменениях.
- **Непрерывная интеграция (CI):** Не настроена. Внедрение CI позволит автоматически проверять качество кода и тесты при каждом изменении.
- **Документация API:** Отсутствует спецификация OpenAPI. Добавление такой документации облегчит интеграцию с внешними системами.

### 1.4 Инфраструктура
- **Единая точка входа:** Отсутствует API Gateway — клиенты обращаются напрямую к разным сервисам.
- **Тестовая среда:** Есть среда для локальной разработки, но нет выделенной тестовой среды, приближенной к продакшену.
- **Мониторинг и отказоустойчивость:** Не реализованы. В текущей версии межсервисные вызовы отсутствуют, поэтому механизмы retry и circuit breaker не требовались, но с развитием архитектуры они станут актуальны.

## 2. Области для улучшения

На основе анализа можно выделить несколько направлений, работа над которыми повысит качество проекта и ускорит разработку новых функций.

| Категория | Область для улучшения | Ожидаемый эффект |
| :--- | :--- | :--- |
| **Код** | Устранение дублирования (shared library) | Снижение объёма кода, упрощение поддержки, единообразие реализаций. |
| **Тестирование** | Добавление модульных и интеграционных тестов | Повышение стабильности, возможность безопасного рефакторинга. |
| **Процессы** | Внедрение CI (GitHub Actions) | Автоматическая проверка качества, раннее обнаружение ошибок. |
| **Архитектура** | Разделение общей БД (Database per Service) | Увеличение независимости сервисов, упрощение масштабирования. |
| **Инфраструктура** | Добавление API Gateway | Централизованное управление трафиком, безопасность, логирование. |
| **Инфраструктура** | Мониторинг и логирование | Прозрачность работы системы, быстрое обнаружение проблем. |
| **Документация** | OpenAPI спецификация | Улучшение взаимодействия с потребителями API. |

## 3. План развития (устранение технического долга)

Предлагаемый план состоит из последовательных шагов, сгруппированных по этапам. Каждый этап приносит ощутимую пользу и готовит почву для следующего.

### Этап 1. Укрепление основы (базовые практики)
1. **Создание общей библиотеки `common-lib`**  
   Вынести повторяющиеся DTO, конфигурации (CORS, валидация) и утилиты для маппинга в отдельный модуль. Подключить его ко всем микросервисам.
2. **Внедрение Continuous Integration (CI)**  
   Настроить GitHub Actions для сборки проекта и запуска тестов при создании Pull Request и пушах в основную ветку.
3. **Добавление тестов для ключевых сценариев**  
   Написать модульные тесты для сервисного слоя и интеграционные тесты для основных REST эндпоинтов (например, создание матча, получение списка игроков).

### Этап 2. Инфраструктурные улучшения
4. **Внедрение API Gateway**  
   Добавить Spring Cloud Gateway как единую точку входа. Настроить маршрутизацию запросов к соответствующим микросервисам. Перенести на Gateway сквозные задачи: CORS, логирование, предварительную аутентификацию.
5. **Улучшение обработки ошибок**  
   Расширить `GlobalExceptionHandler` для возврата детализированных кодов ошибок и локализованных сообщений, используя общую библиотеку.
6. **Настройка базового мониторинга**  
   Интегрировать сбор метрик (Micrometer + Prometheus) и централизованное логирование (ELK или Loki).

### Этап 3. Подготовка к архитектурной эволюции
7. **Анализ зависимостей в базе данных**  
   Документировать, какие таблицы используются несколькими сервисами, и какие внешние ключи существуют. Это станет основой для разделения БД.
8. **Проектирование межсервисного взаимодействия**  
   Определить, как сервисы будут получать данные друг от друга после миграции на отдельные БД (синхронные вызовы через REST/gRPC или асинхронные события).
9. **Внедрение механизмов отказоустойчивости**  
   Добавить Resilience4j (retry, timeout, circuit breaker) для подготовки к появлению сетевых вызовов.

### Этап 4. Эволюция архитектуры (долгосрочная задача)
10. **Поэтапная миграция к Database per Service**  
    Выделять базы данных для сервисов по одному, начиная с наименее связанного (например, Arena). Переписывать запросы, которые ранее обращались к таблицам других сервисов, на вызовы соответствующих API. Миграцию проводить итерационно, с обратной совместимостью.

## 4. Оценка работ

В таблице ниже приведена относительная оценка трудоёмкости каждого мероприятия и возможные риски.

| Мероприятие | Трудоёмкость | Риски |
| :--- | :--- | :--- |
| **1. Создание общей библиотеки** | Низкая | Необходимость версионирования, риск включения лишнего кода. |
| **2. Внедрение CI** | Низкая | Ошибки в конфигурации, необходимость адаптации команды. |
| **3. Добавление тестов** | Средняя | Требует времени на написание, может выявить скрытые дефекты. |
| **4. API Gateway** | Средняя | Появление единой точки отказа, усложнение локальной разработки. |
| **5. Улучшение обработки ошибок** | Низкая | Рисков практически нет. |
| **6. Мониторинг** | Средняя | Сложность настройки агрегации логов из контейнеров. |
| **7. Анализ зависимостей БД** | Низкая | Возможен неполный анализ, если документация неполная. |
| **8. Проектирование межсервисного взаимодействия** | Средняя | Неправильный выбор протокола может привести к задержкам. |
| **9. Отказоустойчивость** | Низкая | Неверные настройки могут ухудшить пользовательский опыт. |
| **10. Миграция к Database per Service** | Высокая | Риск потери данных, длительный простой, сложность отката. |

## 5. Сравнение функциональности и технического состояния

### Функциональность
- Базовая функциональность (CRUD для сущностей Arena, Match, Player, Team) полностью реализована.
- Отсутствуют сквозные нефункциональные возможности: аутентификация, авторизация, версионирование API, документация OpenAPI.

### Техническое состояние
- Архитектура имеет предпосылки для дальнейшего развития: микросервисное деление, современный стек, контейнеризация.
- Зоны роста: дублирование кода, отсутствие тестов и CI, единая база данных, отсутствие единой точки входа.

Обе категории важны: функциональность определяет ценность продукта, а техническое состояние — скорость и стоимость её расширения. Вложения в улучшение технической базы окупаются ускорением разработки новых функций и снижением количества ошибок.

## 6. Рекомендации по корректировке плана разработки

Учитывая текущее состояние, предлагается на ближайшие три спринта сместить фокус с добавления новой функциональности на укрепление технической базы. Это позволит создать надёжный фундамент для будущего роста.

| Спринт | Основные задачи |
| :--- | :--- |
| **Спринт 1** | 1. Создать общую библиотеку `common-lib` и устранить дублирование.<br>2. Настроить CI (GitHub Actions).<br>3. Написать тесты для наиболее критичных сценариев. |
| **Спринт 2** | 1. Внедрить API Gateway.<br>2. Улучшить обработку ошибок, добавить детализированные ответы.<br>3. Настроить базовый мониторинг и логирование. |
| **Спринт 3** | 1. Провести анализ зависимостей в базе данных.<br>2. Спроектировать межсервисное взаимодействие.<br>3. Внедрить механизмы отказоустойчивости (Resilience4j). |

После выполнения этих шагов можно возвращаться к активной разработке новых функций, постепенно реализуя миграцию на отдельные базы данных в рамках реализации новых задач.
